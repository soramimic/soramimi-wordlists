# ADR 00001: 自動更新の対象リストとデータソースの選定基準

- Status: accepted
- Date: 2026-07-05
- Supersedes: none
- Superseded by: 00004-stations-auto-update.md(stations を対象外とした判断のみ)

## Context

単語リストのうち、ネット上の公開情報から機械的に更新できるものは GitHub Actions の定期バッチ(差分があれば PR 作成)で自動更新したい。一方で、このリポジトリのリストには提供者由来のデータ(baseball/football/stations)や人手でキュレーションしたリスト(physicist/sekitsui)が含まれており、何でも自動化するとデータの品質や性格が壊れる。どのリストを自動更新対象とし、ソースを何にするかの基準が必要になった。

## Decision

自動更新の対象は、次の基準をすべて満たすリスト・ソースに限定する。

1. **機械可読な構造化データであること。** HTML のスクレイピングは構造変更で壊れやすいので採用しない。GitHub 上の生 CSV/JSON や認証不要の API を優先する
2. **認証・登録なしで無人取得できること。** バッチにシークレット管理を持ち込まない(REST Countries は v5 から API キー必須になったため不採用とし、その元データである mledoze/countries の生 JSON を使う)
3. **ライセンス上、再配布・クレジット表記が満たせること。** 出典は README のリスト一覧に記載する
4. **既存データの表記・id・人手の補正を壊さない差分方式にできること。** 全件再生成が安全なのは、id と表記が機械的に決まるリスト(pokemon)だけ。それ以外は「新規行の追記+状態列の更新」のみ行う(nations)

この基準で、自動更新対象は **pokemon.csv(PokéAPI)** と **nations.csv(mledoze/countries)** とする。

その他のリストは対象外とする。

- **baseball / football / stations**: 提供者(やきゅうた広場、ヨロスー様、すきやきすきや様)由来のデータで、鮮度・読みの精度が公開ソースより高い。stations の読み付き公開データは駅データ.jp 等の要登録ソースしかなく基準 2 を満たさない
- **physicist / sekitsui**: 「有名どころを人が選ぶ」キュレーションが価値であり、網羅的な自動追記はリストの性格を変えてしまう

**Wikipedia / Wikidata の位置づけ**: 技術的には駅名(一覧記事)、野球選手(カテゴリ)、物理学者(Wikidata の職業クエリ)を取得可能だが、(a) 一覧記事のスクレイピングは基準 1 に反して壊れやすい、(b) 網羅取得はキュレーション性を壊す、(c) 読み(かな)の欠損が多い、という理由で全自動更新には使わない。将来やるなら「候補を機械生成して人が確認して追記する」半自動パイプラインとして別途設計する。

## Consequences

- pokemon / nations は `.github/workflows/update-wordlists.yml` の年次バッチ(1月上旬)で無人更新され、差分は PR レビューで確認できる
- baseball 等の更新は従来どおり README 記載の手順(人手確認込み)で行う
- 新しいリストを自動更新対象に加えるときは、この 4 基準で判断し ADR を追記する
- ソース側の仕様変更(API 廃止等)が起きた場合は、まず同等の構造化データ(元データセットの GitHub リポジトリ等)への乗り換えを検討する
